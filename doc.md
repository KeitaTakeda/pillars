## ⚠️ 2. 必須要件（3つのルール）

### ✅ ルール1: 自分のシステムを使用
- 授業で配布されたサンプル（Calculator等）は使用不可
- 自分のGitHubにある個人開発リポジトリを使用
- または、課題用に新規作成したWebアプリ/APIを使用

### ✅ ルール2: CI/CDツールを3つ統合
- `.github/workflows` に3種類のジョブを作成
- 3つのYAMLファイルまたは1つのYAMLファイル内に3つのジョブを定義

### ✅ ルール3: 最低1つは授業外のCIツール
- **最重要**: Pytest や Linter のみの組み合わせは不可
- 授業で扱っていないツールを必ず1つ以上含める

## 🛠 3. 推奨CI/CD構成

### 基本構成例

| # | 種類 | ツール例 | 用途 |
|---|------|----------|------|
| 1 | 授業内 | Linter | コード整形の自動チェック |
| 2 | 授業内 | Unit Test | テストの自動実行 |
| 3 | **★授業外** | 下記リストから選択 | - |

### ★ 授業外CIツールのオプション

#### 🔒 セキュリティ診断
- **Trivy**: 依存ライブラリの脆弱性チェック
- 実装難易度: ★☆☆

#### ⚡ パフォーマンス計測
- **Lighthouse CI**: フロントエンドのパフォーマンス測定
- **推奨度: 最高** (フロントエンドがある場合)
- 実装難易度: ★☆☆

#### 📄 ドキュメント生成
- **Redocly**: Swagger/OpenAPIからHTML生成
- 実装難易度: ★★☆

#### 🖱️ E2Eテスト
- **Cypress** または **Playwright**: ブラウザ操作の自動化
- 実装難易度: ★★★

## 📝 4. レポート記載項目

各CI/CDツール（3つ）について、以下の4項目を必ず記述してください。

### 4.1 名称と目的 (Name and Purpose)

**記載例**:
```
Lighthouse CI - フロントエンドのパフォーマンス低下を防ぐため
```

### 4.2 設定の詳細 (Configuration Details)

**必須**: GitHub ActionsのYAMLファイルの内容を含める

**記載例**:
````yaml
- name: Run Lighthouse CI
  uses: treosh/lighthouse-ci-action@v9
  with:
    urls: |
      http://localhost:3000
````

### 4.3 成功した証拠 (Screenshots or Logs)

**必須画像**:
- GitHubの「Actions」タブで緑色のチェックマーク✅が表示された画面

**追加推奨**:
- 実行ログの詳細
- 生成されたArtifactのスクリーンショット

### 4.4 導入のメリット (Benefits)

**記載例**:
- 手動チェックの手間が削減された
- 脆弱性に早期に気づけるようになった
- コード品質が向上した
